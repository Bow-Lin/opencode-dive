# 分析规则

## 核心原则

1. 先证据，后抽象。
2. 先入口主线，后边缘情况。
3. 稳定结论放在 `docs/{en,zh}/`，高噪声探索放在 `reports/`。
4. 版本边界必须明确。
5. 未知问题必须保持可见。

## 证据标签

统一使用下面这些标签：

- `Confirmed`：有直接代码证据并且已经引用。
- `Inferred`：结论大概率成立，但来自多个间接信号拼接。
- `Pending verification`：尚未从代码中确认。

## 什么算完成一个模块分析

只有当文档包含下面这些内容时，模块才算真正分析完成：

- 模块职责，
- 具体文件，
- 关键类和函数，
- 初始化方式或入口路径，
- 上下游依赖，
- 至少一条主要控制路径，
- 以及未解决问题。

## 什么算确认一条调用链

只有满足下面条件，调用链才算确认：

- 触发点已识别，
- 第一个具体入口符号已命名，
- 控制流步骤按顺序列出，
- 在相关时注明分支或失败处理，
- 并且每个关键步骤都引用实现它的文件和符号。

## 文档风格

- 优先使用简短的技术段落和表格，不要写空泛总结。
- 使用精确的文件路径和符号名。
- 明确区分事实和解释。
- 避免写出放到任何代码库都成立的通用废话。
- 当中英文文档同时存在时，尽量保持章节结构一致。
